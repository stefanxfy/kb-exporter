# 背景

1、根据业务可以划分多种服务器类型，如媒体融合、流媒体、云录播、语音转写等服务器；

为什么会划分各种服务器？因为服务器上部署的应用对资源有要求，如磁盘空间、磁盘IO、网络带宽等；

2、服务器类型目前基本上和应用组是1对1的关系，一个应用组就可以代表一种服务器。

3、不同类型服务器之间应用运行环境互不影响。


# 方案对比

| 方案对比 | 应用包标签 | 应用资源特征标签 | 应用组标签 |
| --- | --- | --- | --- |
| 核心理念 | 一个应用包一个应用标签，应用包和应用标签，通常1对1，但没有强制要求1对1，如应用包A和应用包B可以设置同一个应用标签。 | 1、资源特征标签（通用）应用对运行环境的要求主要体现在服务器资源上，如CPU、内存、磁盘、IO、网卡、显卡等基于服务器类型（应用组）维度，定义多个资源特征标签资源特征标签属于通用标签，多个应用组可能都具备，如应用组1和应用组2 有相同的资源特征标签，那么这两个应用组中的应用就会混合部署，运行环境互相影响，如CPU、IO资源抢占：业务应用CPU资源超卖严重，request设置很小，limit设置很大，主要为适配多种并发规模（低中高）；k8s容器IO资源未做限制，暂时无法对IO资源进行隔离，需要使用cgroup v2（Linux内核要求4.5以上，稳定5.2），升级k3s版本、jvm sdk兼容。所以应用组和应用组之间运行环境需要隔离，只有资源特征标签还不够，就需要引入业务环境特征标签，2、业务特征标签（隔离）一个应用组代表一种服务器类型，可以表示为一种业务特征，那么这个业务特征标签就可以定义为应用组一个应用包被多个应用组关联，一个应用包就需要选择多个业务特征标签解决多个业务运行环境互相影响的根本问题：资源抢占。 | 一个应用组代表一种服务器类型，给服务器打上应用组标签，应用包里的应用打上应用组标签。 |
| 代码复杂度 | 低✅一个应用包一个应用标签，和应用组/服务器类型没有关系，不需要太多复杂标签定义k8s 标签实现上只需要 nodeSelector 即可 | 高❗️在应用组和应用之外再定义一套特征标签一个应用对应多个特征标签，其中资源特征标签属于弱依赖，业务特征标签属于强依赖，两种标签选择映射到k8s中，实现复杂度高一个应用是否支持打上多个特征标签，标签选择关系为“或”。nodeSelector，只能与的关系，无法实现affinity，可以用matchExpressions 实现多个标签“或”的关系。affinity使用示例展开源码apiVersion: v1
kind: Pod
metadata:
  name: my-pod
spec:
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
        - matchExpressions:
          - key: "A"
            operator: In
            values:
            - "true"
          - key: "B"
            operator: In
            values:
            - "true"
          - key: "C"
            operator: In
            values:
            - "true"
  containers:
  - name: my-container
    image: nginx:latest | 中部署节点时，选择应用组，给节点打上应用组标签鲸云开通应用组时，给其中的应用包打上应用组标签；先后开通多个应用组时，相同应用需要打上多个应用组标签；相同应用（已部署无需升级，只需要改动deployment文件打上应用组标签，重启pod）一个应用需要打上多个应用组标签，标签选择关系为“或”，所以需要用到k8s的亲和性（affinity）。 |
| 实现成本 | 低✅平台和业务都无需做过多兼容，也不需要大范围推动业务整改。 | 高✅1、业务不清楚自己需要多少资源（CPU、内存、IO）合适，目前没有太完善的监控数据支持业务判断。2、IO限制（request、limit）实现难度大，短期无法支持，重要不紧急。3、1和2完成后，还需要推动业务改动应用包资源设置。 | 低✅平台和业务都无需做过多兼容，也不需要大范围推动业务整改。 |
| 扩展性 | 差❗️应用组新增/删除应用（带有应用标签），边缘上同步给相应节点增加/删除应用标签 | 好✅应用组新增/删除应用，不需要变更节点标签。服务器资源共享，最大限度利用资源，从而降低边缘服务器成本，更少的服务器资源，做更多的事情。 | 中✅应用组新增/删除应用，不需要变更节点标签。应用组是一个逻辑概念，具有很强的业务特性，服务器节点标签和应用组概念强耦合，从定义上看属于给业务定制化做功能。 |
| 维护成本 | 低✅一个应用包只有一个应用标签，可自定义输入业务应用标签无需知道其他信息，业务维护成本低 | 高❗️应用包中需要选择多个资源特征标签，多个业务特征标签业务研发需要知道该应用包需要哪些资源，以及被哪些应用组依赖，业务维护成本高 | 低✅业务研发在维护应用包 时，无需关心打什么标签；部署应用组时，由鲸云自动给其中应用包中的应用打上应用组标签。 |


总结：

1、基于应用组（服务器）设置标签，太具有业务特性和定制化，所以不采用。

2、抽象出一套通用的资源特征标签，实现上以及概念上不耦合任何业务，做到真正的通用，但是实现难度大，周期长，目前并不具备这样的条件；基于现有业务特性以及交付成本，选择方案：“应用包标签”；

3、从长远看，“应用资源特征标签”是最终实现目标，它把k3s集群的优点发挥的最好，比如，所有服务器之间资源共享，降低服务器成本（不需要那么多的服务器，这也是一个市场优势）；需要为该方案做前置准备和长期规划：

- 建立健全应用资源监控采集，供业务观测，并能辅助业务统计出应用资源使用合理数值；
- 边缘k3s支持IO限制（request、limit），并辅助业务设置合理的IO request 和 limit。
